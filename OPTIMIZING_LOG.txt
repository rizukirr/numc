[2026-03-13] F32 ASM 8x K-Unroll + Gemmsup OMP Parallelism
================================================================================
Commit: (pending)
Files:  src/intrinsics/gemm_avx2.h, src/intrinsics/gemmsup_avx2.h,
        src/matmul/matmul.c

Problem:
  1. The f32 ASM kernel had only 4x K-unroll with 1 prefetch (A only), while
     the f64 kernel had 8x K-unroll with 3 prefetches. This caused more loop
     overhead and L1 cache misses on the B panel for f32 at 256x256+.
  2. 128x128 sat exactly at the gemmsup boundary (m*k*n < 128^3), falling to
     packed GEMM where packing overhead + OMP scheduling variance caused
     unstable results (28us to 136us across runs).
  3. Gemmsup kernels had no OMP parallelism, making them slower than numpy's
     multi-threaded BLAS at 128x128.

Changes:
  1. [gemm_avx2.h] Added GEMM_F32_ASM_K_ITER macro — extracts the 20-instr
     single K-iteration (matching f64's GEMM_F64_ASM_K_ITER pattern)
  2. [gemm_avx2.h] Rewrote GCC/Clang ASM K-loop: shr $2 -> shr $3 (8x unroll)
     - 3 prefetches per 8 iterations:
       * prefetcht0 384(%[ap])  — A panel ahead
       * prefetcht0 512(%[bp])  — B panel ahead (was missing entirely)
       * prefetcht0 768(%[ap])  — A panel further ahead
     - k_left mask: and $3 -> and $7
  3. [gemm_avx2.h] Updated MSVC intrinsics fallback: kc/4 -> kc/8, added
     B prefetch and second A prefetch to match ASM path
  4. [matmul.c] Changed gemmsup threshold from < to <= so 128x128 (exactly
     128^3 FLOPs) routes to the unpacked gemmsup path instead of packed GEMM
  5. [gemmsup_avx2.h] Added OMP parallel for to f32 and f64 gemmsup dispatch
     loops (parallelizes over MR-row tiles, threshold: M*K*N > 1<<20)

Before (vs NumPy/OpenBLAS, multi-threaded):
  f32 64x64:     1.04x
  f32 128x128:   0.97x (but unstable: 0.14x-0.97x due to OMP variance)
  f32 256x256:   0.77x
  f32 512x512:   0.89x
  f32 1024x1024: 0.73-0.80x

After (representative results, OMP_PROC_BIND=close):
  f32 64x64:     1.00x  (stable)
  f32 128x128:   0.80x  (single bench run; best observed 1.62x)
  f32 256x256:   1.03x  (+34% improvement)
  f32 512x512:   0.69-1.23x  (high variance from system load)
  f32 1024x1024: 0.87-1.10x  (high variance from system load)

  f64 64x64:     0.99x
  f64 128x128:   1.54x
  f64 256x256:   1.03-2.60x  (depends on system load)
  f64 512x512:   0.70-1.51x
  f64 1024x1024: 0.68-0.87x

  Single-threaded verification (eliminates OMP variance):
  f32 128x128:   numc 44us vs numpy 43us → 1.02x (parity)

Note: Benchmark results have high run-to-run variance due to system load
  during testing. The 8x K-unroll change primarily benefits 256x256+ sizes
  by reducing loop overhead and improving L1 prefetching. The gemmsup OMP
  change eliminates the packed GEMM scheduling variance for 128x128.

Problem:
  Scalar comparisons (eq/gt/lt/ge/le_scalar) relied on auto-vectorization
  via `#pragma omp simd` in the generic DEFINE_BINARY_KERNEL macro's
  `sb == 0` path. No dedicated SIMD intrinsics existed for scalar compares.
  Integer types were especially slow because the compiler couldn't optimize
  the unsigned XOR-bias trick or certain comparison patterns.

  f32 scalar comparisons: ~0.60x vs NumPy
  f64 scalar comparisons: ~0.80x vs NumPy
  Integer types: varied, many below 1.0x

Changes:
  1. [compare_scalar_avx2.h] New — 50 functions (10 types × 5 ops).
     Float: _mm256_cmp_ps/pd with IMM predicates, AND with 1.0.
     Signed int: _mm256_cmpeq/cmpgt_epiW, AND with 1.
     Unsigned int: XOR sign-bit bias + signed compare (pre-computed
     biased scalar outside loop for efficiency).

  2. [compare_scalar_avx512.h] New — 50 functions.
     Float: _mm512_cmp_ps/pd_mask + _mm512_mask_blend.
     Signed int: _mm512_cmp_epiW_mask with _MM_CMPINT predicates +
     _mm512_maskz_set1. Native 64-bit compare (no emulation).
     Unsigned int: _mm512_cmp_epuW_mask — native unsigned compare.

  3. [compare_scalar_neon.h] New — 50 functions.
     All types have native NEON comparisons (vceqq, vcgtq, vcgeq,
     vcltq, vcleq for s/u/f types). Float uses vbslq for 0/1 select.

  4. [compare_scalar_sve.h] New — 50 functions.
     Predicated loops with svwhilelt. Scalar-immediate compare
     variants (svcmpeq_n_*) avoid explicit broadcast. svsel for 0/1.

  5. [compare_scalar_rvv.h] New — 50 functions.
     Vector-scalar compare intrinsics (_vx/_vf) for efficiency.
     LMUL=m4. vmerge/vfmerge for 0/1 conversion.

  6. [compare.c] Added CmpScKern function pointer tables per op,
     ISA-selected via CMPSC() macro. New DEFINE_CMP_SCALAR_WITH_SIMD
     macro intercepts contiguous scalar comparisons and dispatches to
     SIMD kernels. OMP parallelization for large arrays (>512KB per
     thread) splits work into chunks, each processed by SIMD kernel.
     Falls back to generic _scalar_op for non-contiguous arrays.

Before (vs NumPy, 1M elements):

  f64 scalar eq/gt/lt/ge/le:  ~0.80x
  i8:   ~1.0x    u8:  ~1.0x

  i64:  ~0.90x   u64: ~0.90x

[2026-03-15] AVX2 Edge Packing Cleanup (Maskload/Zero-Fill Path)
================================================================================
Commit: pending
Files:  src/intrinsics/gemm_avx2.h

Problem:
  AVX2 packed GEMM edge-tile packing still had avoidable overhead in remainder
  branches:
  1. Extra branch in B-pack tails (`if (rem > 8)` / `if (rem > 4)`) per K-iter
     despite already using masked loads.
  2. A-pack tails built vectors with multiple scalar conditionals per column,

[2026-03-14] OMP Hoisting + Parallel B Packing + AVX2 NC/Alignment
================================================================================
Commit: 594fc5e
Files:  src/intrinsics/gemm_avx2.h, src/intrinsics/gemm_avx512.h,
        src/intrinsics/gemm_neon.h, src/intrinsics/gemm_sve.h,
        src/intrinsics/gemm_rvv.h

Problem:
  1. Repeated OMP fork/join — `#pragma omp parallel` was inside the PC loop,
     forking/joining threads 4× for K=1024. Each fork costs 20-50μs with high
     jitter, causing benchmark variance (f32 512: 0.69-1.23x, f32 1024:
     0.87-1.10x).
  2. Serial B packing — all threads idle while one thread packs the entire B
     panel sequentially.
  3. AVX2 GEMM_F32_NC was 2048, while all other ISAs already used 4080
     (matching BLIS Haswell configuration).

Changes:
  1. [All 5 ISA files] Hoisted `#pragma omp parallel` outside the PC loop.
     Threads now fork once per JC iteration (outer N-blocking loop) instead
     of once per PC iteration (inner K-blocking loop). This eliminates 3 of
     every 4 fork/join pairs for K=1024 (KC=256, 4 PC iterations).
     OMP threshold changed from m_dim*kc*nc (per-PC volume) to m_dim*k_dim*nc
     (total volume) since the decision is now per-JC, not per-PC.

  2. [All 5 ISA files] Added per-strip B packing helper functions:
     - AVX2:  _gemm_pack_b_strip_f32/_f64 (uses _mm256 intrinsics)
     - AVX512: _gemm_pack_b_strip_f32/_f64 (uses _mm512 intrinsics)
     - NEON:  _gemm_pack_b_strip_f32/_f64 (uses vst1q/vld1q intrinsics)
     - SVE:   _gemm_pack_b_strip_f32_sve/_f64_sve (runtime nr parameter)
     - RVV:   _gemm_pack_b_strip_f32_rvv/_f64_rvv (runtime nr parameter)
     B packing is now parallelized via `#pragma omp for schedule(static)`
     over NR-wide column strips, with an implicit barrier before the compute
     phase. Each strip function handles full-width SIMD path (nr_pack == NR)
     and scalar edge path (nr_pack < NR) with zero-padding.

  3. [gemm_avx2.h] Changed GEMM_F32_NC from 2048 to 4080, matching BLIS
     Haswell and all other ISAs (NEON/SVE/RVV already used 4080).

  4. [gemm_avx2.h] Changed 4 ASM K-loop alignment points from .p2align 4
     (16-byte) to .p2align 5 (32-byte) for Haswell DSB optimization.

  Structure change (all ISAs, f32 and f64):
    Before:
      for jc:
        for pc:              ← K-blocking loop
          pack_b()           ← SERIAL, all threads idle
          #pragma omp parallel  ← FORK (20-50μs)
            omp for tasks
          }                  ← JOIN (barrier)
    After:
      for jc:

Optimization #11: Batched Pointer Advance in ASM Micro-Kernels (NEGATIVE RESULT)
================================================================================

Problem:
  numc's 8× unrolled K-loop had per-iteration pointer advances (add rA, stride;
  add rB, stride) — 16 add instructions per 8-iter body. BLIS batches pointer

Optimization #14: BLIS-Style SIMD Transpose A-Packing (f32 + f64)
================================================================================

Problem:
  A-packing converts row-major source A into column-panel format (MR×KC). The
  previous implementation used scalar loads for each element of each row, then
  stored to the pack buffer. With MR=6 and KC up to 512, this is a lot of scalar
  memory traffic. BLIS Haswell kernels use SIMD transpose-and-pack: load multiple
  K columns at once from each row, transpose in-register, and store as contiguous
  MR-strided panels. This reduces load instructions and enables full-width SIMD
  stores.

Solution:
  Replaced scalar A-packing loops with SIMD transpose packing in gemm_avx2.h:

  f64 (gemm_pack_a_f64): 4× K-unroll.
    - Load 4 doubles from each of 6 rows (4 × _mm256_loadu_pd per row, 24 total).
    - Transpose rows 0-3 using _mm256_unpacklo/hi_pd + _mm256_permute2f128_pd
      to get 4 column vectors, each containing {row0[k], row1[k], row2[k], row3[k]}.
    - Transpose rows 4-5 using _mm256_unpacklo/hi_pd + _mm256_extractf128_pd
      to get 4 pairs of {row4[k], row5[k]}.
    - Store each column as 4 doubles (rows 0-3) + 2 doubles (rows 4-5) = MR=6.
    - Scalar cleanup for kc % 4 remainder columns.

  f32 (gemm_pack_a_f32): 8× K-unroll.
    - Load 8 floats from each of 6 rows (6 × _mm256_loadu_ps).
    - Transpose rows 0-3 using _mm256_unpacklo/hi_ps + _mm256_shuffle_ps(0x4E) +
      _mm256_blend_ps(0xCC/0x33) to get 8 column vectors for rows 0-3.
    - Transpose rows 4-5 using _mm256_unpacklo/hi_ps + 128-bit extract +
      _mm_castps_pd + _mm_storel/h_pd for paired stores.
    - Store each column as 4 floats (rows 0-3) + 2 floats (rows 4-5) = MR=6.
    - Scalar cleanup for kc % 8 remainder columns.

  Edge cases (ir < mc, i.e., partial MR panels) still use scalar packing.

Result (same-session A/B comparison: git stash → baseline × 5 runs, restore × 5 runs):

  Best-of-5 numc absolute times (us):
